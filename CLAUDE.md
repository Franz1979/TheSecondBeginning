# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

TheSecondBeginning is a 2D top-down world-simulation game built in **Godot 4.6** (GDScript, Forward+ renderer, Jolt physics, D3D12 on Windows). There is no external build system, package manager, or test framework — the project is opened and run directly in the Godot editor.

## Commands

- **Run the game**: open the project in the Godot 4.6 editor and press F5 (runs `run/main_scene` from `project.godot`), or run headless from the CLI if a Godot binary is available: `godot4 --path .`
- **Run a specific scene**: F6 in the editor, or `godot4 --path . scenes/game/GameScene.tscn`
- There are no lint, test, or build commands configured in this repo (no addons/, no test framework).
- `.gd` scripts each have a paired `.uid` file — these are Godot-managed (resource UIDs); don't hand-edit them, and always keep them alongside their script when moving/renaming files.

## Architecture

### Scene flow
`MainMenu` → `NewGameMenu` / `MapEditorMenu` → `GameScene` / `MapEditorScene`. Navigation is done via `get_tree().change_scene_to_file(...)`; state is *not* passed as scene params but through the `GameSettings` autoload (`scripts/core/GameSettings.gd`), which holds `selected_map_type` (`"random"`, `"island"`, `"saved"`, `"empty"`), `selected_map_file`, `selected_save_file`, and `selected_macro_cell_x`/`selected_macro_cell_y`, plus the `user://maps/` and `user://saves/` directories. Menu scripts set these fields before switching scenes; `GameScene`/`MapEditorScene` read them in `_ready()`/`_load_world()` to decide what to construct or load.
- `GameScene` → `MacroCellScene`: double-clicking a cell in `GameScene` (detected in `_unhandled_input` via `InputEventMouseButton.double_click`) stores the clicked cell's coordinates in `GameSettings.selected_macro_cell_x/y` and switches to `scenes/game/MacroCellScene.tscn` — a zoomed-in detail view of that one macro cell's internal microcells (`MacroCellState.TOTAL_SPACE` = 10000 = a 100x100 grid, same dimensions as the macro world). As of now this is a structural placeholder only: `MacroCellScene` doesn't yet read `selected_macro_cell_x/y` or the clicked cell's actual data — it just reuses `World`/`WorldRenderer`/`InitialResourceSetupService` to generate and render its own disconnected, freshly-generated 100x100 grid, with a minimal sidebar (Save stub, "Torna al Mondo" back button) built the same way as `GameScene`'s (right-docked opaque `Sidebar` `PanelContainer`, reusing `CameraController` as-is — its `_get_map_rect()` lookup of `CanvasLayer/Sidebar` works unchanged since the node path matches). Deliberately deferred, not yet designed: a real per-macro-cell microcell data model, an advance-year control for this scene (year must stay a single coherent clock shared with `GameScene`, not a second independent counter — this is tangled up with an eventual switch from year-granularity to day-granularity plus time-speed controls), and safely returning to `GameScene` without losing unsaved changes (today "Torna al Mondo" just calls `change_scene_to_file` back to `GameScene`, which always reloads from `GameSettings`/disk in `_load_world()` — any edits since the last save are lost, same as it would be today without this feature).

### World model
- `GameTypes.gd` (`scripts/core/GameTypes.gd`) is the single source of truth for all simulation enums (`TerrainBase`, `WaterType`, `Biome`, `CoastType`, `RiverShape`, `WorldObjectType`, `ResourceType`, `SuccessionLevel`, `NaturalEventType`). Add new terrain/resource/event categories here first.
- `World` (`scripts/world/world.gd`) owns a flat 100x100 grid (`WIDTH`/`HEIGHT` consts) of two parallel arrays indexed by `y * WIDTH + x`:
  - `cells: Array[MacroCellData]` — static terrain/geography (terrain base, water, biome, coast, river shape).
  - `cell_states: Array[MacroCellState]` — dynamic per-cell simulation state (resource quantities, "dedicated space" per resource type, river space). `TOTAL_SPACE` (10000) is the microcell budget each macro cell has to allocate across resources.
  - Always fetch cells via `world.get_cell_at(x, y)` / `get_cell_state_at(x, y)`, never index the arrays directly — these validate coordinates and index consistency.
- World generation is pluggable: `RandomMapGenerator` and `PresetMapGenerator` (island preset) both take a `World` and mutate it in place; `World.generate_empty_world()` dispatches to the right generator based on `GameSettings.selected_map_type`. `WorldProcessors` holds post-processing passes (e.g. coast derivation from adjacency).

### Resource simulation
Resource behavior (stone, trees, grass, shrubs, more to come per `GameTypes.WorldObjectType`) is data-driven via Godot `Resource` subclasses stored as `.tres` files, not hardcoded. There are currently three renewable resources — `TREE`, `GRASS`, `SHRUB` — that share the same growth/migration/encroachment logic and only differ by their `.tres` values.
- `ResourceDensityRules` / `ResourceGrowthRules` (`scripts/core/`) are `@export`-driven Resource classes with base values plus multipliers per terrain/biome/coast. `ResourceGrowthRules` also carries a `Succession` group (`succession_level`, `encroachment_rate`, `max_encroachment_per_year`) used by encroachment.
- Data files live at `res://data/resource_density/{type_name}_density.tres` and `res://data/resource_growth/{type_name}_growth.tres`, where `{type_name}` is the lowercased `GameTypes.WorldObjectType` enum key (e.g. `tree_growth.tres`). `ResourceCalculator` (`scripts/core/ResourceCalculator.gd`) resolves and loads these by convention — adding a new resource type means adding the enum value in `GameTypes` and dropping in matching `.tres` files, no code changes needed in the calculator.
- `GameTypes.SuccessionLevel` (`FORAGE` → `SHRUB` → `TREE`) models ecological succession: renewable resources have a `succession_level` in their growth rules, and higher-succession types can encroach on lower ones occupying the same cell (e.g. trees overtaking shrubs, shrubs overtaking grass) when they have unmet growth surplus. `ResourceCalculator.get_types_ordered_by_succession()` sorts a set of resource types ascending by `succession_level` and is the shared ordering used by growth, encroachment, mortality, and migration so processing is always lowest-succession-first.
- Per-year simulation is split into services invoked from `GameScene._on_advance_year_pressed()`, in order: `ResourceGrowthService.grow_resources()` (logistic growth of dedicated space/quantity per cell, scaled by each cell's `get_active_growth_multiplier()` — see Natural events below) → `ResourceEncroachmentService.encroach_resources()` (lets each renewable type consume space from weaker-succession types present in the same cell, farthest succession level first, capped by `max_encroachment_per_year` and `encroachment_rate`-derived efficiency; growth surplus it can't place is returned as a `{Vector2i: {WorldObjectType: float}}` leftover-surplus dictionary) → `ResourceMigrationService.compute_transfers(world, leftover_surplus)` (computes, but does not yet apply, transfers of that leftover surplus into neighboring cells N/S/E/W, probabilistic) → `ResourceMortalityService.apply_mortality()` (density-dependent self-thinning: reuses each resource's own `get_growth_rate()` as the mortality base, scaled by a step multiplier on the macro cell's whole-cell fill ratio — `dedicated_space` of all resources plus river space over `TOTAL_SPACE` — negligible below 50% full, rising in three bands up to 90%+; frees space back to `empty_space`) → `ResourceMigrationService.apply_transfers(world, transfers)` (applies the previously computed transfers, now that mortality has freed room in target cells, capped per-resource by `get_empty_space()` at application time) → `NaturalEventService.trigger_events()` (see Natural events below). Mortality is applied *after* transfers are computed but *before* they're applied so freshly-freed space is claimed by incoming migrants this same year rather than by the source cell's own growth next year. `InitialResourceSetupService` seeds starting resources when a new game/world is created.
- All of these "service" classes (`*Service.gd`, `RefCounted`, no autoload) are stateless — instantiate with `.new()` and call their single entry method per use; this is the established pattern for any new simulation logic.

### Natural events
- `GameTypes.NaturalEventType` (`NONE`, `FIRE`, `DROUGHT`, `SEA_FLOOD`, ...) enumerates event categories, mirroring the `WorldObjectType`/growth-rules convention: rules live in data files, code stays generic per event type. Each rules `.tres`'s `event_name` is a `tr()` localization key (e.g. `"event_fire"`), not a literal display string — same convention as the `tr()`-wrapped UI labels (see UI scripts below); nothing currently calls `tr(rules.event_name)` since no UI surfaces event names to the player yet, but the data is kept key-shaped so that display point can be added later without touching the `.tres` files again.
- `NaturalEventRules` (`scripts/core/NaturalEventRules.gd`, `@export`-driven `Resource`) holds the fields shared by every event type — trigger probability, `requires_coastal_center` (generic geographic constraint, see below), intensity weights/radius, per-succession-level fragility, a destroyed-cells budget per intensity tier, and a flat growth-rate multiplier/duration applied to affected cells (a booster >1 for post-fire regrowth, a penalty <1 for drought/salinization stress — same fields, opposite direction). Event-specific rule sets subclass it when they need extra/different fields: `FireEventRules` adds spread probability and max spread iterations for fire's flood-fill, and keeps the base class's flat `post_event_growth_duration_years` (fire's recovery is always a fixed length regardless of intensity); `DroughtEventRules` and `SeaFloodEventRules` instead each add their own `post_event_growth_duration_by_intensity` (an array — a more severe event's growth penalty lasts longer), used instead of the base class's flat duration field (which stays at its unused default on those two, harmless since nothing reads it there — set to `0` in their `.tres` as a reminder it's dead for that type). Data files live at `res://data/natural_events/{type_name}_event.tres`, loaded by convention through `NaturalEventCalculator.get_event_rules()` — same lookup-by-enum-name pattern as `ResourceCalculator`.
- `NaturalEventService.trigger_events(world, year)` runs once per year, last in `GameScene._on_advance_year_pressed()`'s pipeline (see order above): it ticks down each cell's `active_growth_bonuses` first, then for every type in `REGISTERED_EVENT_TYPES` rolls a Poisson-like event count from `base_probability_per_year`, and for each event picks a center cell (uniformly random across the whole map, unless `rules.requires_coastal_center` is set — then `_pick_random_coastal_cell()` restricts the pick to cells with `coast_type != NONE`, i.e. adjacent to sea), an intensity tier, and a radius, producing a `NaturalEventInstance`. Adding a new event type means adding the enum value, a rules subclass if needed, a matching `.tres`, an effect service, and a case in `NaturalEventService._apply_event_effects()` — plus adding it to `REGISTERED_EVENT_TYPES` to activate it.
- `VegetationDestructionService` (`scripts/core/VegetationDestructionService.gd`) is the shared "distribute a destroyed-space budget across affected cells" routine used by any event that destroys vegetation: splits a total budget evenly per cell, then within each cell across present `GRASS`/`SHRUB`/`TREE` weighted by `fragility_weight_by_succession_level × dedicated_space` (more fragile/succession-heavy and denser stands lose more). `FireEventEffectService`, `DroughtEventEffectService`, and `SeaFloodEventEffectService` all call into it rather than duplicating the weighting logic.
- `FireEventEffectService` ("tempesta di fulmini"/lightning-strike fire) is the effect handler for `FIRE`: from the event's center it flood-fills outward N/S/E/W up to `max_spread_iterations`, bounded by the event radius, spreading probabilistically (`spread_probability_to_neighbors`) into adjacent cells that have flammable vegetation (any of `GRASS`/`SHRUB`/`TREE` with `dedicated_space > 0`), then calls `VegetationDestructionService` with the intensity tier's destroyed-cells budget. Every affected cell (burned or not, as long as touched by the event) gets a temporary growth-rate *boost* (`post_event_growth_multiplier` > 1) registered via `MacroCellState.register_growth_bonus()`, simulating post-fire regrowth vigor.
- `DroughtEventEffectService` (`DROUGHT`) and `SeaFloodEventEffectService` (`SEA_FLOOD`) share the same shape, no subclassing between them (each is its own effect service, small enough not to be worth a common base): no probabilistic spread — the affected area is every land cell (`terrain_base != WATER`) within Manhattan distance `radius` of the event center, filled directly (for `SEA_FLOOD` the center itself is already coast-constrained via `requires_coastal_center`). Both call `VegetationDestructionService` for an instant destruction budget, then register a growth-rate *penalty* (`post_event_growth_multiplier` < 1) for a duration resolved from their own `post_event_growth_duration_by_intensity[intensity_index]` — drought models a dry spell (mild multiplier, `[1,2,3,4,6]`-ish year range); sea flood models soil salinization (harsher multiplier, longer `[2,3,5,7,10]`-ish year range, since salt takes longer to leach out than a dry spell to break). A river-flood variant (silt deposition → growth *boost* instead of penalty) is planned but not yet implemented.
- `MacroCellState.active_growth_bonuses` (`NaturalEventType -> {multiplier, years_remaining, total_duration}`) is persisted in saves (`GameSaveService`/`GameLoadService`, key `active_growth_bonuses`) so bonuses survive save/load and keep ticking down correctly. Multiple event types can have simultaneously-active bonuses on the same cell (multipliers compound via `get_active_growth_multiplier()`). `get_active_growth_multiplier()` skips any bonus at `years_remaining <= 0`, and `tick_growth_bonuses()` only erases a bonus once `years_remaining` goes *negative* — this one-cycle lag is deliberate: it keeps the bonus (and its on-map marker) visible for exactly as many years as it actually affected growth (1 "fresh" year + N "recovering" years, matching `total_duration`), rather than the marker vanishing a year before the growth effect actually ended. `WorldRenderer.RENDERED_EVENT_TYPES` draws a distinct marker per type on top of the resource overlay (red/orange X for `FIRE`, brown/ocher circle for `DROUGHT`, dark/pale-blue square for `SEA_FLOOD`; the two-tone split is "fresh" — the event's own year — vs "recovering" — subsequent years until the bonus expires).

### Rendering & input
- `WorldRenderer` (`scripts/world/worldRenderer.gd`, `Node2D`) draws the whole grid immediate-mode in `_draw()` — terrain color per cell, river shapes, and a resource-quantity overlay — keyed off `CELL_SIZE = 10` px per macro cell. Call `queue_redraw()` after mutating world/cell state to refresh.
- Cell picking/input is handled by `CellSelectorController` (base: click-to-select, emits `cell_selected(cell, state)`), extended by `MapEditorController` which adds terrain "painting" (brush + drag-paint) and keeps river/coast adjacency consistent as you paint (`_update_river_shape_around`, `_update_coast_type_around`).
- `CameraController` (`scripts/camera/CameraController.gd`) is a plain WASD/arrow-pan + scroll-zoom `Camera2D`, shared by game and editor scenes.

### Persistence
Save/load is hand-rolled JSON (no Godot `ResourceSaver`), with parallel service pairs:
- `WorldSaveService`/`WorldLoadService` — map-only, `file_type: "world_map"`, used by the map editor.
- `GameSaveService`/`GameLoadService` — full save including `GameData` (currently just `year`) and world + cell_states, `file_type: "game_save"`.
Both loaders validate `file_type` before parsing and reject/`push_error` on mismatch. When extending `MacroCellData`/`MacroCellState`/`GameData` with new fields, update both the matching save and load service together, and use `.get(key, default)` on load for new/optional fields to stay compatible with older save files.

### UI scripts
Menu/HUD scripts (`scripts/scenes/*.gd`, `scripts/game/GameScene.gd`, `scripts/world/MacroCellInfoPanel.gd`) follow a consistent pattern: `@onready` node refs matched to the paired `.tscn` structure, `tr()`-wrapped button labels (localization keys, though no translation CSV exists yet), and signal wiring done in `_ready()`.
